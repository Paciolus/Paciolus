# Anomaly Coverage Map

> **Authority:** This is the authoritative mapping from synthetic anomaly type
> to production detector, detection threshold, and user-visible output route.
> Every anomaly in the registry MUST have a row here. The validation test
> `test_coverage_map.py` enforces this.
>
> **Source:** AUDIT-06 Phase 3 findings (2026-03-20).

| Anomaly type | Registry generator | Production detector | Diagnostic(s) surfaced in | Minimum detectable threshold | Coverage status | Notes |
|---|---|---|---|---|---|---|
| `round_number` | `generators/round_number.py` · `RoundNumberGenerator` | `audit/rules/rounding.py` · `detect_rounding_anomalies()` | RPT-01, RPT-21 | `ROUNDING_MIN_AMOUNT = 10000.0` + divisible by one of `ROUNDING_PATTERNS` (10k/50k/100k). Tier 1 accounts are suppressed entirely. Constant in `classification_rules.py`. | COVERED | Three-tier framework (Sprint 536–538) suppresses fixed assets, depreciation, payroll, etc. False positives reduced but blind spot exists for Tier 1 accounts that are genuinely anomalous. |
| `suspense_account` | `generators/suspense_account.py` · `SuspenseAccountGenerator` | `audit/rules/suspense.py` · `detect_suspense_accounts()` | RPT-01, RPT-21 | `SUSPENSE_CONFIDENCE_THRESHOLD = 0.60` — sum of matched keyword weights must reach 0.60. Keywords and weights defined in `classification_rules.py :: SUSPENSE_KEYWORDS`. | COVERED | Blind spot: zero-balance suspense accounts are skipped (`abs(net) < BALANCE_TOLERANCE`). Naming-only anomalies (suspense account with $0 balance) are not flagged — this is by design (non-zero balance is the audit concern). |
| `abnormal_balance` | `generators/abnormal_balance.py` · `AbnormalBalanceGenerator` | `audit/rules/balance.py` · `detect_abnormal_balances_streaming()` | RPT-01, RPT-21 | `BALANCE_TOLERANCE = 0.01` (from `shared/monetary.py`) — accounts with `abs(net) < 0.01` are skipped. Classification confidence thresholds: `CONFIDENCE_HIGH = 0.7`, `CONFIDENCE_MEDIUM = 0.4`. `AccountCategory.UNKNOWN` accounts are never flagged. | COVERED | Detection quality is sensitive to classification accuracy. If the heuristic classifier miscategorizes an account type, the expected-vs-actual balance direction check may produce false positives or miss true violations. CSV-provided account types bypass heuristic classification. |
| `weekend_posting` | `generators/weekend_posting.py` · `WeekendPostingGenerator` | `audit/rules/relationships.py` · `detect_intercompany_imbalances()` | RPT-01, RPT-03, RPT-21 | No single centralized threshold constant. Detection requires: (1) account name matches `INTERCOMPANY_KEYWORDS` from `classification_rules.py`, (2) `abs(net) >= BALANCE_TOLERANCE` (0.01), (3) counterparty extracted from account name via separator heuristic, (4) counterparty group net ≠ 0. Threshold is emergent from keyword match + balance tolerance + counterparty extraction logic. | PARTIAL | Threshold not centralized — detection boundary is determined by the combination of keyword matching in `classification_rules.py :: INTERCOMPANY_KEYWORDS` and the counterparty extraction heuristic in `relationships.py :: _extract_counterparty()`. Centralization recommended. Account names without recognizable separators (" — ", " - ", etc.) will not have counterparties extracted, causing intercompany pairs to be missed. |
| `missing_offset` | `generators/missing_offset.py` · `MissingOffsetGenerator` | Balance check in `audit/streaming_auditor.py` (total debits vs total credits) | RPT-01, RPT-21 | No single centralized threshold constant. Detection relies on `total_debits != total_credits` (balance check). The tolerance is `BALANCE_TOLERANCE = 0.01` applied to the overall TB balance comparison. Any orphan entry > $0.01 that unbalances the TB will be detected. | PARTIAL | Detection is indirect — the engine flags the TB as `balanced=False` but does not identify which specific entry caused the imbalance. The orphan entry itself is not individually flagged; the user sees that the TB does not balance. Field assumptions (that a single orphan debit without offset will unbalance the TB) are correct for single-entity TBs but may not hold for pre-consolidated multi-entity uploads. Centralization of the detection path recommended. |
| `duplicate_entry` | `generators/duplicate_entry.py` · `DuplicateEntryGenerator` | Balance check in `audit/streaming_auditor.py` (TB-level); `je_testing_engine.py :: test_duplicate_entries()` (JE tool); `ap_testing_engine.py :: test_exact_duplicate_payments()` + `test_fuzzy_duplicate_payments()` (AP tool) | RPT-01, RPT-03, RPT-21 (TB-level); JE and AP tool reports (tool-specific) | TB-level: `BALANCE_TOLERANCE = 0.01` (detects imbalance caused by duplicates, not the duplicates themselves). JE: exact key match on `(date, account, debit, credit, description)`. AP-T1: exact key match on `(vendor, invoice, amount, date)`. AP-T6: same vendor + amount ±`duplicate_tolerance` (default 0.01) within `duplicate_days_window` (default 30 days). | GAP / PARTIAL | Fragmented across three detection contexts with no unified contract. TB anomaly rules do NOT detect duplicate rows — they only detect the balance impact. JE and AP tools detect duplicates at the transaction level but only when the user runs those specific tools. A duplicate entry that preserves TB balance (e.g., duplicated pair of offsetting entries) would be invisible at the TB level. See `DUPLICATE_ENTRY_CONTRACT.md` for per-tool contract details. |
