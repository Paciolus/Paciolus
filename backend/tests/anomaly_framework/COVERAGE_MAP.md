# Anomaly Coverage Map

> **Authority:** This is the authoritative mapping from synthetic anomaly type
> to production detector, detection threshold, and user-visible output route.
> Every anomaly in the registry MUST have a row here. The validation test
> `test_coverage_map.py` enforces this.
>
> **Source:** AUDIT-06 Phase 3 findings (2026-03-20).

## Blind Spots — Documented Future Work

The following anomaly categories are neither detected by production code nor represented in the synthetic framework. They require new detector logic and are out of scope for the current remediation cycle.

| Category | Description | Risk | Notes |
|---|---|---|---|
| Channel stuffing | Abnormal revenue spikes near period-end paired with high returns in subsequent period | HIGH | Requires multi-period data; single-TB analysis insufficient |
| Cookie-jar reserves | Excessive or understated reserves manipulated across periods | HIGH | Requires reserve trend analysis across periods |
| Bill-and-hold | Revenue recognized before delivery; deferred revenue anomalies | HIGH | Requires contract/delivery data not available in TB |
| Premature revenue recognition | Revenue recorded before performance obligations are satisfied | HIGH | Requires ASC 606 contract analysis; partially addressed by Revenue Testing tool |
| Improper capitalization | Operating expenses capitalized as assets to inflate earnings | MEDIUM | Requires account-level trend analysis and reclassification detection |
| Unrecorded liabilities | Obligations present but not reflected in the TB | MEDIUM | Inherently undetectable from TB alone; requires external confirmation |
| Round-trip related-party transactions | Circular flows through related entities to inflate revenue/assets | HIGH | Requires entity-level flow analysis and related-party mapping |
